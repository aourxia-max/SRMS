# Task004 楼栋与房源验收记录

## 范围

本任务以 `docs/requirements-freeze-v1.md`、`docs/database-design.md` 和交接文档的 Task004 定义为准，仅交付楼栋与房源基础资料维护：楼栋/房源 CRUD、房态历史、状态筛选、详情编辑和权限边界。

未提前实现承租人、合同、账单、收款、驾驶舱或其他后续任务功能。

## 已实现

- Prisma migration `20260722130000_task004_rooms`：`buildings`、`rooms`、`room_status_histories`，含冻结的房源类型、面积、装修、用途、业主展示资料、房态与状态变化时间。
- 后端接口 `/api/properties`：已登录用户可读取楼栋、房源与状态历史；仅 `SUPER_ADMIN`、`ADMIN` 可新建、编辑、变更房态和软删除房源。
- 后端始终根据楼栋编号与房号生成 `full_house_no`；修改楼栋编号、房号或所属楼栋时同步重算完整房号。
- 房态首次建档和每次实际变化均在同一事务中写入 `room_status_histories`，并保存当前操作人。
- 前端中文楼栋/房源页：楼栋和房源新增/编辑、完整房号、面积和业主展示、房态筛选、状态历史和删除确认；访客仅有查看权限。

## 手工验收步骤

1. 使用超级管理员或管理员登录，进入“楼栋与房源”。
2. 新增楼栋和房源，确认完整房号由系统生成，且面积、装修、用途、业主资料能保存。
3. 修改房态后打开“历史”，确认新旧状态、原因和时间有记录。
4. 用访客登录，确认可查看但不显示新增、编辑、删除和房态修改操作；直接调用写接口也应被后端拒绝。

## 已知边界

“存在未结束合同的房源不能改为待出售、已出售或停用”的校验依赖 Task006 合同模型；该模型尚未创建，因此本任务没有虚构合同数据或规则。Task006 创建合同后必须在房态变更接口中补上此强制校验。
